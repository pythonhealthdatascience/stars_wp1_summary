5.2 Structure of code and scripts

Model is provided in a ‚Äúrunnable‚Äù format
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚úÖ ‚ùå

Huang et al. (2019): The code provided was to run the model as part of a web application, but the paper was not focused on this application, but instead on specific model scenarios, so I had to extract the model code and transform it into a format that was ‚Äúrunnable‚Äù as an R script/notebook.

Model is designed to be run programmatically
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚úÖ üü°

Shoaib and Ramamohan (2021): Model was designed so that you directly change parameters in the script itself. This meant you couldn‚Äôt run multiple versions of the model using the same script. It also meant hidden errors were more likely (e.g. if miss changing a parameter somewhere, or input the wrong parameters and don‚Äôt realise).

Huang et al. (2019): Model was set up as a function, with many of the required parameters already set as inputs to that function.

Lim et al. (2020): Structure of provided code (model largely in functions) made it easier when making changes to run it programmatically. However, it wasn‚Äôt all in functions, and the model itself was designed to run a single scenario, so I did need to change to a function that can call to easily vary parameters and run scenarios programmatically.

Don‚Äôt hard code parameters that you will want to change for scenario analyses
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚ùå ‚ùå

Shoaib and Ramamohan (2021): Parameters were often hard-coded, and so it took time to spot where these were and correctly adjust them to be modifiable inputs.

Lim et al. (2020): Several parameters were hard coded

Avoid large amounts of code duplication
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå

Shoaib and Ramamohan (2021): The model often contained very similar code before or after warm-up. This has the potential to introduce mistakes - with a suspected (although unconfirmed) mistake being that the lower boundary for the doctor consultation times in configuration 1 differed before and after warm-up.

Include sufficient comments in the code
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚úÖ

Huang et al. (2019): Would have benefitted from more comments, as it took some time to decipher and ensure I have correctly understood code as it uses lots of abbreviations.

Lim et al. (2020): Lots of comments in the code (including doc-string-style comments at the start of functions) that aided understanding of how it worked.

5.3 Run time and memory usage

Quicker models are easier to work with

I have not evaluated like as a criteria, as a long run time is not inherently a bad thing. However, I definitely found that the run time of models had a big impact on how easy it was to reproduce results as longer run times meant it was tricky (or even impossible) to run in the first place, or tricky to re-run.

The studies where I made adjustments were:

  • Shoaib and Ramamohan (2021): Add parallel processing and ran fewer replications
  • Lim et al. (2020): Add parallel processing
For slow models, it‚Äôs very important to state the expected run time
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå

For long models with no statement, it can take a while to realise that it‚Äôs not an error in the code or anything, but actually just a long run time! And hard to know how long to expect, and whether it is without the capacities of your machine and so on.

For computationally expensive models, state memory usage, and potentially include alternatives for running it on machines without the capacity for it
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)

5.4 Parameters, scenarios and outputs

Provide code for all scenarios
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚ùå ‚ùå

Shoaib and Ramamohan (2021): There were several instances where it took quite a while to understand how and where to modify the code in order to run scenarios (e.g. no arrivals, transferring admin work, reducing doctor intervention in deliveries).

Huang et al. (2019): Set up a notebook to programmatically run the model scenarios. It took alot of work to modify and write code that could run the scenarios, and I often made mistakes in my interpretation for the implementation of scenarios, which could be avoided if code for those scenarios was provided.

Lim et al. (2020): Several parameters or scenarios were not incorporated in the code, and had to be added (e.g. with conditional logic to skip or change code run, removing hard-coding, adding parameters to existing).

All the required outputs are calculated/provided
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚ùå

Shoaib and Ramamohan (2021): Had to add some outputs and calculations (e.g. proportion of childbirth cases referred, standard deviation)

Lim et al. (2020): The model script provided was only set up to provide results from days 7, 14 and 21. The figures require daily results, so I needed to modify the code to output that.

Include correct parameters in the script (even if just for one scenario)
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå

Huang et al. (2019): The baseline model in the script did not match the baseline model in the paper, so had to modify parameters.

Provide all the required parameters
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚ùå ‚ùå

Shoaib and Ramamohan (2021): Some parameters that could not be calculated were not provided - ie. what consultation boundaries to use when mean length of doctor consultation was 2.5 minutes

Huang et al. (2019): In this case, patient arrivals and resource numbers were listed in the paper, and there were several discprepancies between this and the provided code. However, for many of the model parameters like length of appointment, these were not mentioned in the paper, and so it was not possible to confirm whether or not those were correct.

Lim et al. (2020): For Figure 5, had to guess the value for staff_per_shift.

If not provided in the script, then clearly present all parameters in the paper
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚úÖ

Shoaib and Ramamohan (2021): Although there was a scenario table, this did not include all the parameters I would need to change. It was more challenging to identify parameters that were only described in the body of the article. There were also some discrepancies in parameters between the main text of the article, and the tables and figures. Some scenarios were quite ambiguous/unclear from their description in the text, and I initially misunderstood the required parameters for the scenarios.

If will need to process parameters, provide required calculations
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚úÖ

Shoaib and Ramamohan (2021): It was unclear how to estimate inter-arrival time

Huang et al. (2019): The calculations for inter-arrival times were provided in the code, and the inputs to the code were the number of arrivals, as reported in the paper, and so making it easy to compare those parameters and check if numbers were correct or not.

5.6 Seeds

Use seeds to control stochasticity
Shoaib and Ramamohan (2021) Huang et al. (2019) Lim et al. (2020) Kim et al. (2021) Anagnostou et al. (2022) Johnson et al. (2021)
‚ùå ‚ùå ‚ùå

Shoaib and Ramamohan (2021): The lack of seeds wasn‚Äôt actually a barrier to the reproduction though due to the replication number. I later add seeds so my results could be reproduced, and found that the ease of setting seeds with salabim was a greater facilitator to the work. I only had to change one or two lines of code to then get consistent results between runs (unlike other simulation software like SimPy where you have to consider the use of seeds by different sampling functions). Moreover, by default, salabim would have set a seed (although overridden by original authors to enable them to run replications).

Huang et al. (2019): It would have been beneficial to include seeds, as there was a fair amount of variability, so with seeds I could then I could be sure that my results do not differ from the original simply due to randomness.

Lim et al. (2020): The results obtained looked very similar to the original article, with minimal differences that I felt to be within the expected variation from the model stochasticity. However, if seeds had been present, we would have been able to say with certainty. I did not feel I needed to add seeds during the reproduction to get the same results.
